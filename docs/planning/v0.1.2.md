# v0.1.2 Planning

v0.1.2 is planned as a small engineering-quality release. It should improve
reviewer confidence and maintenance without adding large algorithms.

## Goals

- Keep the stable public API explicit and documented.
- Keep point-cloud IO errors path-aware and easy to debug.
- Keep portfolio and benchmark artifacts schema-checkable.
- Keep the portfolio pipeline internally modular while preserving import and
  CLI compatibility.
- Keep release gates manual, transparent, and non-publishing.
- Keep repository audit output useful when `gh` is available and friendly when
  it is not.

## Planned Work

- API: continue testing `TaskResult.to_dict()` and `TaskResult.to_json()` error
  envelopes.
- IO: extend bad-input coverage for `.bin`, `.off`, `.ply`, `.pcd`,
  `.xyz`, and `.txt` readers.
- Schema: validate required keys and numeric ranges for portfolio, benchmark,
  and release manifest JSON.
- Pipeline: keep inputs, metrics, figures, reports, HTML, and runner logic in
  smaller modules.
- Reviewer workflows: keep `verify-core`, `verify-portfolio`,
  `verify-benchmarks`, and `verify-release-candidate` documented.
- Audit: summarize version, tag, release, issue, CI, coverage, generated guard,
  optional dependencies, and limitations.

## Out Of Scope

- No full nonlinear GICP optimizer.
- No real KITTI benchmark report.
- No SLAM backend.
- No CUDA acceleration.
- No PointNet training release.
- Downloading or committing real KITTI, ModelNet, SemanticKITTI, or nuScenes
  data.

## Release Discipline

- Do not rewrite the v0.1.1 tag or release.
- Do not commit generated artifacts under `outputs/`, `results/`,
  `examples/demo_data/`, `benchmark_results/`, `htmlcov/`, `dist/`, or
  `build/`.
- Do not describe synthetic fixtures as real-data validation.
- Keep coverage gate at a stable value rather than the current exact coverage.
